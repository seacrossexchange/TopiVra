# TopiVra 下一阶段执行方案（运营级落地与可交付）

> 目标：让下一次执行可以**直接开工**，并以“完整运营、真实落地、可交付验收”为唯一标准推进到 100%。

---

## 1. 交付目标与验收口径

### 1.1 总目标
在现有代码基础上，完成 TopiVra 的三角色（买家 / 卖家 / 管理员）运营级闭环，达到：

- 功能真实可用（非样式页、非占位页）
- 核心链路稳定可回归（无首次点击空白、无刷新后才恢复）
- 管理后台可运营（配置可理解、动作可执行、结果可追踪）
- 五语种可用且专业（无技术字符暴露）
- 测试、构建、回归可证明

### 1.2 完成定义（DoD）
仅当以下全部满足，才算“100% 完成”：

1. 三角色主链路全部通过（买家下单、卖家上架与履约、管理员审核与运营）
2. 历史重复页完成保留/合并/下线裁决并落库（代码与路由一致）
3. 商品体系支持账号类与软件类有效上架（含分类、国家策略、交付方式）
4. 博客系统支持卖家投稿、管理员编写、管理员审核发布
5. 教程阅读权限支持公开/登录/会员/打赏（至少 MVP）
6. 广告位功能具备完整投放逻辑（创建→审核→投放→统计）
7. 主题切换全局一致，三套布局体验统一
8. i18n 五语种关键页面无技术字符与硬编码污染
9. 反站外引流具备最小可用风控能力（拦截+审核+处罚记录）
10. `client` / `server` 构建通过，核心测试通过，关键回归通过

---

## 2. 当前状态复盘（作为下次执行起点）

### 2.1 已有基础（可复用）
- 三角色结构已具备初步收敛基础
- 前端 build / test 曾多次通过
- 主题、路由、登录、产品页问题已有过修复尝试
- 已有 E2E 与多类测试脚本，可用于回归

### 2.2 未闭环风险（必须先清）
- “宣称完成”与“用户实际反馈”存在偏差（同类问题反复出现）
- 历史页残留与双轨路由风险未最终裁决
- 主题全局一致性需要组件级全站验收
- 首次点击空白问题需建立稳定复现与压测回归
- 管理端部分模块存在“看起来有，实际逻辑不清”风险（典型：广告位）
- i18n 仍有技术字符/术语不统一问题

---

## 3. 执行原则（强制）

1. **证据优先**：每个完成项必须有代码、截图/录屏、测试结果
2. **主线优先**：所有优化都归入“100% 可运营主线”，不做无关扩展
3. **先 P0 后体验**：先清稳定性与闭环，再做视觉/文案 polish
4. **一次裁决，避免反复**：历史页保留/合并/下线要定版，不重复摇摆
5. **运营视角验收**：每项功能回答“谁用、怎么进、怎么做、结果是什么”

---

## 4. 分阶段执行计划（可直接开工）

## 阶段 A（P0 稳定性封口，1-2 天）
**目标**：消灭“首次点击空白 / URL 变化页面不渲染 / 首次登录异常”等阻塞项。

### A1. 路由与首屏稳定性专项
- 建立复现矩阵：
  - 首页 → 产品详情
  - 首页 → 登录
  - 未登录访问受保护页
  - 三角色后台首次进入
- 增加运行时日志（开发/测试环境）：
  - 路由切换开始/完成
  - 页面挂载/卸载
  - 错误边界触发
- 清理冲突机制（SEO/head、auth bootstrap、lazy/Suspense）

### A2. 认证初始化专项
- 统一登录入口到 store/action，禁止页面私有登录链路
- hydration 完成前统一 loading 策略
- token-only 状态补全用户信息策略固定化

### A3. 主题全局一致性专项
- ThemeProvider 唯一真源
- antd token 与 CSS 变量同源
- Main/Admin/Seller 布局统一接入
- 组件巡检：Menu/Card/Table/Modal/Drawer/Dropdown

**阶段验收**：连续 50 次关键导航无空白、无白屏、无首击失败。

---

## 阶段 B（三角色结构最终裁决，1 天）
**目标**：彻底消除历史残留与重复入口。

### B1. 路由与页面裁决清单（必须成文）
对候选历史页逐项定案：
- 保留
- 合并
- 下线

### B2. 代码落地
- 删除下线路由与无效导航
- 合并页保留重定向与兼容入口（短期）
- 清理未引用组件与死代码

### B3. 角色入口专业化
- 头像区变为“身份入口 + 后台直达”
- 多角色账号可一键进入对应后台

**阶段验收**：路由树唯一、无重复主入口、角色边界清晰。

---

## 阶段 C（卖家商品体系产品化，2 天）
**目标**：让卖家可以真实上架并交付账号类 + 软件类商品。

### C1. 商品模型升级
- 商品类型：账号类 / 软件类 / 数字资源类
- 分类：一级+二级
- 国家属性策略：
  - 可无国家
  - 单国家
  - 多国家

### C2. 上架表单动态化
- 通用字段 + 类型专属字段
- 软件支持交付：zip/exe/link/key
- 发布前规则校验，支持草稿保存

### C3. 前后台一致性
- 后台字段前台可展示
- 无展示映射的字段不得上线

**阶段验收**：完成 2 个样本上架（账号类 1、软件类 1）并前台可购买/可交付。

---

## 阶段 D（管理员中台真功能化，2 天）
**目标**：把“看不懂逻辑”的模块改为可运营闭环。

### D1. 广告位系统重构（优先）
必须明确：
- 广告位定义（位置）
- 素材规范
- 跳转目标
- 投放规则（时段/语言/地区）
- 审核机制
- 上下线机制
- 统计口径（曝光/点击）

### D2. 审核流统一
- 商品、博客、广告统一状态机
- 审核记录与驳回原因可追踪

**阶段验收**：运营可独立完成“创建广告→审核→投放→查看效果”。

---

## 阶段 E（博客 + 权限阅读 + 变现，2 天）
**目标**：博客成为真实内容运营能力。

### E1. 角色与审核
- 卖家可投稿、管理员可编写
- 统一管理员审核发布

### E2. 阅读权限 MVP
- 公开
- 登录可读
- 会员可读
- 打赏解锁
- 会员或打赏可读

### E3. 前台交互
- 未达权限显示摘要+引导
- 解锁后阅读全文

**阶段验收**：3 篇文章跑通（公开/会员/打赏各 1 篇）。

---

## 阶段 F（i18n + 风控 + 总回归，2 天）
**目标**：达到“可上线交付”标准。

### F1. i18n 五语种收口
- 清理硬编码与技术字符
- 术语表统一（三角色一致）
- 关键页面逐页验收

### F2. 反站外引流 MVP
- 输入拦截（标题/描述/博客/工单）
- 风险词规则
- 举报与处罚记录
- 管理台最小可用审核面板

### F3. 全量回归
- client build/test
- server build/test
- e2e 关键链路

**阶段验收**：P0/P1 缺陷清零，形成可交付报告。

## 8. 风险与回退策略

### 8.1 主要风险
- 修复范围扩大导致连锁回归
- 旧页面被隐藏引用导致回退失败
- i18n 修改影响大量 UI 文案
- 风控规则过严影响正常发布

### 8.2 回退机制
- 关键模块分支隔离（路由/主题/商品/博客/风控）
- 每阶段完成后独立回归再合并
- 保留短期重定向与兼容策略（带截止日期）
